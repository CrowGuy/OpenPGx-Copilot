# 02 Domain Model

## 1. Purpose

This document defines the core domain model for OpenPGx Copilot v0.1.

OpenPGx Copilot is a public-data-driven genetic interpretation and pharmacogenomics decision-support prototype.

Version 0.1 is a wellness-first educational proof of concept. It uses low-risk genetic trait examples to validate the core architecture:

```text
synthetic rsID-genotype input
→ curated rule matching
→ rule-bound evidence lookup
→ educational explanation
→ safety validation
```

The domain model must support v0.1 wellness trait interpretation while remaining compatible with future pharmacogenomics modules.

## 2. Domain Model Principles

The domain model follows these principles:

```text
1. Runtime interpretation uses active curated records only.
2. Compiler-generated records are generated (non-active) until a release path activates them: generated → validated → diff/exception reviewed → release-approved → active.
3. Evidence must be explicitly linked to active rules.
4. User-facing explanations must be generated from matched rules and bound evidence.
5. LLMs may explain but must not decide domain logic.
6. Association is not the same as recommendation.
7. Wellness v0.1 outputs are educational only.
8. Future PGx modules require additional allele, diplotype, phenotype, and drug-rule models.
9. OpenPGx reviews the system, release diffs, and exceptions; it does not require per-rule manual review for every generated record.
```

The governance floor still requires mandatory human sign-off for: release approval, certified-compiler (re)certification, changes to the trusted-source set, and per-item review of exceptions and safety-sensitive diffs. These controls are equivalent-or-stronger than per-rule review, not looser: nothing generated by an uncertified compiler or from an untrusted source can ever be activated, and every release still passes strong validation, a diff, resolved exception reviews, and an explicit release approval before serving.

## 4. Active vs Candidate Records

A central distinction in OpenPGx Copilot is:

```text
candidate record ≠ active record
```

Candidate records may be generated by source adapters, compilers, reviewer tools, or manual drafts.

Active records are records that have passed the release path (generated → validated → diff/exception reviewed → release-approved → active) and are released for runtime use. Records generated by an uncertified compiler or from an untrusted source can never be activated.

Runtime components must use only active records.

```text
Candidate records:
- EvidenceCandidateRecord
- RuleCandidateRecord
- ReviewCard
- DiffReport

Active records:
- EvidenceRecord
- RuleRecord
- RuleSetRelease
```

Candidate records must not produce user-facing interpretations.

## 5. SourceRecord

A `SourceRecord` represents the original source from which evidence, marker metadata, or candidate records were derived.

It answers:

```text
Where did this information come from?
```

Example:

```yaml
id: source_aldh2_rs671_paper_001
source_type: peer_reviewed_article
title: "..."
url: "..."
pmid: "..."
doi: "..."
retrieved_at: "2026-07-02"
raw_payload_path: "data/raw/aldh2/source_001.json"
sha256: "..."
license_note: "..."
```

Supported v0.1 source types:

```text
gwas_catalog_association
dbsnp_variant_record
ncbi_variation_service_record
pubmed_metadata
peer_reviewed_article
open_targets_record
clinvar_record
population_frequency_record
manual_curated_source
```

Required fields:

```text
id
source_type
retrieved_at
```

Recommended fields:

```text
title
url
pmid
doi
raw_payload_path
sha256
license_note
source_version
```

A `SourceRecord` must not directly produce a user-facing interpretation.

## 6. MarkerRecord

A `MarkerRecord` represents a genetic marker used by the system.

In v0.1, this is usually an rsID-level marker.

Example:

```yaml
id: marker_rs671
marker_type: rsid
rsid: rs671
gene_context:
  - ALDH2
assembly: GRCh38
normalized_variant:
  representation: SPDI
  value: "..."
observed_alleles:
  - G
  - A
source_refs:
  - source_dbsnp_rs671_001
status: active
```

A `MarkerRecord` answers:

```text
What is this marker?
Is this rsID valid?
What alleles or variant representations are known for it?
What gene context is associated with it?
```

v0.1 marker records may be used for:

```text
1. rsID validation.
2. Genotype format validation.
3. Allele compatibility checks.
4. Source traceability.
5. Future normalization.
```

A `MarkerRecord` does not itself define a trait interpretation.

## 7. EvidenceCandidateRecord

An `EvidenceCandidateRecord` is a non-active record generated from structured or semi-structured sources.

It may come from:

```text
GWAS Catalog
dbSNP / NCBI Variation Services
PubMed metadata
Open Targets
ClinVar
manual evidence discovery
```

Example:

```yaml
id: evidence_candidate_aldh2_rs671_gwas_001
status: candidate
domain: wellness_trait
source_refs:
  - source_gwas_rs671_001
  - source_pubmed_rs671_001
gene: ALDH2
markers:
  - rs671
trait: alcohol_metabolism
reported_trait: "alcohol-related trait"
association:
  p_value: "..."
  effect_allele: "..."
  risk_allele: "..."
  effect_size: "..."
population:
  ancestry: "..."
  sample_size: "..."
candidate_summary: >
  Candidate association record linking rs671 with an alcohol-related trait.
candidate_limitations:
  - "Association must pass the release path (validation, diff/exception review, release approval) before use."
  - "Effect allele does not automatically define genotype-level interpretation."
candidate_allowed_use:
  - reviewer_workflow
candidate_disallowed_use:
  - user_facing_interpretation
  - medical_advice
  - nutrition_advice
  - medication_advice
requires_human_review:
  - "Confirm trait mapping."
  - "Confirm effect direction."
  - "Confirm population note."
  - "Confirm no disease-risk overclaim."
```

An `EvidenceCandidateRecord` may support review workflows.

It must not be used by the runtime explanation path.

## 8. EvidenceRecord

An `EvidenceRecord` is an active curated evidence object that can be referenced by active rules.

It answers:

```text
What evidence supports this rule?
What can this evidence be used for?
What must it not be used for?
```

Example:

```yaml
id: evidence_aldh2_rs671_001
status: active
domain: wellness_trait
source_refs:
  - source_aldh2_rs671_paper_001
gene: ALDH2
markers:
  - rs671
trait: alcohol_metabolism
summary: >
  ALDH2 rs671 is associated with reduced ALDH2 enzyme activity
  and altered acetaldehyde metabolism.
population_note: "Association is especially relevant in East Asian populations."
limitations:
  - "Genotype is only one factor."
  - "Actual response may be affected by health status, environment, lifestyle, and other genetic factors."
allowed_use:
  - educational_explanation
disallowed_use:
  - medical_advice
  - nutrition_advice
  - fitness_advice
  - supplement_advice
  - medication_advice
  - diagnosis
  - disease_risk_prediction
```

Required fields:

```text
id
status
domain
source_refs
summary
allowed_use
disallowed_use
limitations
```

For v0.1, active evidence must allow:

```text
educational_explanation
```

and must disallow:

```text
medical_advice
nutrition_advice
fitness_advice
supplement_advice
medication_advice
diagnosis
disease_risk_prediction
```

## 9. EvidenceRecord Retrieval Fields

Even though v0.1 runtime does not use open-ended retrieval for user-facing interpretation, evidence records should include retrieval-oriented metadata for future reviewer workflows.

Example:

```yaml
retrieval:
  search_text: >
    ALDH2 rs671 alcohol metabolism acetaldehyde East Asian population
  keywords:
    - ALDH2
    - rs671
    - alcohol metabolism
    - acetaldehyde
  aliases:
    - ALDH2 Glu504Lys
  bm25_fields:
    - gene
    - markers
    - trait
    - summary
  embedding_fields:
    - summary
    - limitations
    - population_note
```

Structured entities:

```yaml
entities:
  genes:
    - ALDH2
  markers:
    - rs671
  traits:
    - alcohol_metabolism
  drugs: []
  phenotypes: []
  populations:
    - East Asian
```

These fields support future evidence viewer retrieval, not runtime interpretation decisions.

## 10. RuleCandidateRecord

A `RuleCandidateRecord` is a non-active candidate rule generated by a compiler or drafted for review.

Example:

```yaml
id: rule_candidate_aldh2_rs671_alcohol_metabolism_001
status: candidate
domain: wellness_trait
gene: ALDH2
trait: alcohol_metabolism
markers:
  - rsid: rs671
    genotype_pattern_candidate: "AG|AA"
candidate_basis:
  - evidence_candidate_aldh2_rs671_gwas_001
compiler_notes:
  - "Candidate generated from GWAS Catalog association and dbSNP marker validation."
requires_human_review:
  - "Confirm genotype pattern."
  - "Confirm effect direction."
  - "Confirm population note."
  - "Confirm no disease-risk claim."
candidate_disallowed_use:
  - user_facing_interpretation
  - medical_advice
  - nutrition_advice
  - medication_advice
```

A `RuleCandidateRecord` may help reviewers create active rules.

It must not be loaded by the runtime Rule Engine.

## 11. RuleRecord

A `RuleRecord` is an active curated rule used by the runtime Rule Engine.

It answers:

```text
What input pattern triggers what interpretation?
```

Example:

```yaml
id: aldh2_rs671_alcohol_metabolism
status: active
domain: wellness_trait
gene: ALDH2
trait: alcohol_metabolism
markers:
  - rsid: rs671
    genotype_pattern: "AG|AA"
interpretation_level: educational
result_code: reduced_aldh2_activity_association
evidence_refs:
  - evidence_aldh2_rs671_001
required_safety_message: wellness_educational
```

Required fields:

```text
id
status
domain
markers
interpretation_level
result_code
evidence_refs
required_safety_message
```

A `RuleRecord` must not be active if:

```text
1. status is not active.
2. evidence_refs is empty.
3. Any evidence_ref cannot be resolved.
4. Any referenced evidence is inactive.
5. Referenced evidence disallows educational explanation.
6. genotype_pattern is invalid.
7. required_safety_message is missing.
```

## 12. RuleRecord Interpretation Level

v0.1 supports only:

```text
educational
```

Future versions may add additional levels, but not in v0.1.

Disallowed v0.1 interpretation levels:

```text
clinical
diagnostic
prescriptive
therapeutic
nutritional
fitness
supplement
```

## 13. TraitInterpretationResult

A `TraitInterpretationResult` is the structured output generated after a rule match.

Example:

```json
{
  "rule_id": "aldh2_rs671_alcohol_metabolism",
  "domain": "wellness_trait",
  "trait": "alcohol_metabolism",
  "gene": "ALDH2",
  "matched_marker": {
    "rsid": "rs671",
    "genotype": "AG"
  },
  "interpretation_level": "educational",
  "result_code": "reduced_aldh2_activity_association",
  "summary": "This genotype is associated with reduced ALDH2 enzyme activity in some populations.",
  "limitations": [
    "Genotype is only one factor.",
    "Actual response may be affected by environment, health status, lifestyle, and other genetic factors."
  ],
  "evidence_refs": [
    "evidence_aldh2_rs671_001"
  ],
  "safety_message": "This explanation is for educational purposes only and should not be used as medical, nutrition, fitness, supplement, or medication advice."
}
```

This object is generated from:

```text
matched RuleRecord
+ resolved EvidenceRecord
+ safety template
```

It must not be generated directly by an LLM.

## 14. RoutingDecision

A `RoutingDecision` represents the structured output of the request adapter or future Gene Domain Intent Router.

Example:

```json
{
  "request_id": "req-001",
  "source": "internal_request_adapter",
  "intent": "trait_interpretation",
  "domain": "genetic_wellness",
  "target_system": "openpgx_copilot",
  "target_module": "wellness_traits",
  "response_mode": "educational_explanation",
  "safety_route": "allowed",
  "extracted_entities": {
    "markers": [
      {
        "rsid": "rs671",
        "genotype": "AG"
      }
    ],
    "traits": [],
    "genes": [],
    "drugs": [],
    "phenotypes": []
  },
  "blocked_outputs": [
    "medical_advice",
    "nutrition_advice",
    "fitness_advice",
    "supplement_advice",
    "medication_advice",
    "diagnosis",
    "disease_risk_prediction"
  ],
  "confidence": 0.95
}
```

The RoutingDecision may influence:

```text
intent
target module
response mode
safety route
entity extraction
```

It must not decide:

```text
genotype-to-trait mapping
evidence strength
rule match
result_code
interpretation_level
```

OpenPGx Copilot must validate the RoutingDecision before execution.

## 15. SafetyRoute

A `SafetyRoute` defines how the request should be handled from a safety perspective.

Supported v0.1 safety routes:

```text
allowed
downgrade_to_educational
refuse_personalized_advice
refuse_medical_advice
refuse_nutrition_advice
refuse_fitness_advice
refuse_supplement_advice
refuse_medication_advice
refuse_disease_risk
unsupported_input
out_of_scope
needs_clarification
```

A safety route may be proposed by a router, but OpenPGx Copilot must validate it.

## 16. ExplanationRecord

An `ExplanationRecord` represents the user-facing explanation generated from structured outputs.

Example:

```yaml
id: explanation_req_001
language: en
response_mode: educational_explanation
source_result_refs:
  - aldh2_rs671_alcohol_metabolism
evidence_refs:
  - evidence_aldh2_rs671_001
body: >
  This result is associated with reduced ALDH2 enzyme activity in some populations.
  Genotype is only one factor, and actual response may depend on health status,
  environment, lifestyle, and other genetic factors.
safety_message: >
  This explanation is for educational purposes only and should not be used as
  medical, nutrition, fitness, supplement, or medication advice.
validation_status: passed
```

An `ExplanationRecord` must be produced from bounded inputs:

```text
TraitInterpretationResult
EvidenceRecord
SourceRecord metadata
SafetyRoute
Safety templates
```

It must not introduce new domain claims.

## 17. OutputSafetyValidationResult

An `OutputSafetyValidationResult` records whether the final output passed safety checks.

`status` is `passed` when `errors` is empty, otherwise `failed`. Each check maps to a fixed error code; the full check-to-code taxonomy is defined in 07_llm_explanation_layer.md section 32.1.

Passed example:

```yaml
id: output_validation_req_001
request_id: req-001
status: passed
checks:
  required_safety_message: passed
  evidence_refs_present: passed
  evidence_refs_in_allowed_set: passed
  source_refs_in_allowed_set: passed
  no_medical_advice: passed
  no_nutrition_advice: passed
  no_fitness_advice: passed
  no_supplement_advice: passed
  no_medication_advice: passed
  no_diagnosis: passed
  no_disease_risk_prediction: passed
  no_deterministic_overclaim: passed
  limitations_present: passed
  no_candidate_records_used: passed
  safety_route_respected: passed
errors: []
```

Failed example:

```yaml
id: output_validation_req_014
request_id: req-014
status: failed
checks:
  no_disease_risk_prediction: failed
  evidence_refs_in_allowed_set: failed
errors:
  - code: disease_risk_prediction_detected
    field: body
    message: "Output contained a disease-risk prediction."
  - code: invented_evidence_reference
    field: evidence_refs
    message: "evidence_ref not present in the bounded context."
```

If validation fails, the response must not be returned; the system applies the fixed fallback order (deterministic template, then refusal) defined in 07_llm_explanation_layer.md section 32.4. v0.1 does not retry the LLM.

## 18. RuleSetRelease

A `RuleSetRelease` represents a versioned set of active rules and evidence.

Example:

```yaml
rule_set_id: wellness_rules_v0_1_2026_07
status: active
domain: wellness_trait
created_at: "2026-07-07"
source_snapshot_refs:
  - gwas_catalog_snapshot_2026_07
  - dbsnp_snapshot_2026_07
evidence_record_version: "0.1"
rule_record_version: "0.1"
review_status: approved
test_status: passed
active_rule_refs:
  - aldh2_rs671_alcohol_metabolism
active_evidence_refs:
  - evidence_aldh2_rs671_001
```

A `RuleSetRelease` must not be active unless:

```text
1. All rules are schema-valid.
2. All evidence_refs resolve.
3. All referenced evidence records are active.
4. Safety validation passes.
5. Regression tests pass.
6. Review status is approved.
```

## 18a. RuntimeReleaseManifest

A `RuleSetRelease` and an `EvidenceSetRelease` are versioned separately, so a rule set could reference evidence IDs that happen to resolve in a different, separately reviewed set. A `RuntimeReleaseManifest` removes that ambiguity: it pins one coherent, reviewed, and tested bundle, and runtime loads only the combination the manifest names.

Example:

```yaml
release_id: wellness_release_v0_1_2026_07
status: active
created_at: "2026-07-07"
rule_set_id: wellness_rules_v0_1_2026_07
evidence_set_id: wellness_evidence_v0_1_2026_07
source_snapshot_refs:
  - gwas_catalog_snapshot_2026_07
  - dbsnp_snapshot_2026_07
schema_version: "0.1"
normalization_version: "0.1"
review_status: approved
test_status: passed
compiler_certification_ref: compiler_cert_wellness_v0_1_2026_07
compilation_run_ref: compilation_run_wellness_2026_07_01
source_trust_profile_refs:
  - trusted_source_gwas_catalog_2026_07
  - trusted_source_dbsnp_2026_07
validation_report_refs:
  - validation_report_wellness_2026_07_01
diff_report_ref: diff_report_wellness_v0_1_2026_07
exception_review_refs:
  - exception_review_wellness_2026_07_01
release_approval_ref: release_approval_wellness_v0_1_2026_07
```

Runtime loading rules:

```text
1. Runtime loads active records only through the active RuntimeReleaseManifest,
   and only the rule_set_id + evidence_set_id it pins. Loading a rule set or
   evidence set by ID without the manifest is not allowed.
2. The rule set and evidence set must agree with the manifest on schema_version,
   normalization_version, and source_snapshot_refs.
3. Every evidence_ref in the pinned rule set must resolve within the pinned
   evidence set, not any other set.
4. review_status and test_status must be approved/passed on the manifest and on
   both pinned sets.
5. An active manifest must carry certified-compiler + trusted-source provenance,
   validation reports, a diff report, resolved exception reviews, and a release
   approval. Concretely: compiler_certification_ref must resolve to a
   CertifiedCompiler with certification_status certified; source_trust_profile_refs
   and compilation_run_ref must resolve; validation_report_refs must all resolve
   with status pass; diff_report_ref must resolve; every exception_review_ref must
   resolve with status approved; and release_approval_ref must resolve.
6. Output from an uncertified compiler or an untrusted source can never be
   activated, regardless of other fields.
7. Any mismatch or missing/unresolved provenance is a strict failure: the service
   fails to load (see the strict loading rule in 05_rule_engine_spec.md section 23).
```

Content hashes/checksums of the pinned sets are future work; v0.1 pins by ID and enforces version consistency.

## 18b. TrustedSourceProfile

A `TrustedSourceProfile` declares that a specific public source, at a specific version/snapshot, is trusted as compiler input. Changing the trusted-source set (adding, removing, or re-scoping a source) requires human approval.

Example:

```yaml
id: trusted_source_gwas_catalog_2026_07
source_identity: gwas_catalog
snapshot_id: gwas_catalog_snapshot_2026_07
license: "EMBL-EBI terms"
schema: gwas_catalog_association_v1
allowed_domain:
  - wellness_trait
allowed_use:
  - compiler_input
trust_status: trusted
approved_by: governance_board
approved_at: "2026-07-01"
```

Only sources with an approved `TrustedSourceProfile` may feed a `CompilationRun`. Output derived from a source without such a profile can never be activated.

## 18c. CertifiedCompiler

A `CertifiedCompiler` records the identity, certification status, and input scope of a compiler. Output from an uncertified compiler can never be activated.

Example:

```yaml
compiler_id: wellness_rule_compiler
version: "0.1.0"
allowed_input_source_profiles:
  - trusted_source_gwas_catalog_2026_07
  - trusted_source_dbsnp_2026_07
determinism_contract: "Same source snapshots + same version produce byte-identical output hashes."
certification_status: certified
certification_suite_version: "cert_suite_0.1"
certified_by: governance_board
certified_at: "2026-07-01"
```

Rules:

```text
1. certification_status must be certified for the compiler's output to be activated.
2. Uncertified compiler output can never be activated.
3. A compiler may only consume sources listed in allowed_input_source_profiles,
   each of which must be an approved TrustedSourceProfile.
4. (Re)certification requires human approval.
```

## 18d. CompilationRun

A `CompilationRun` records one deterministic run of a `CertifiedCompiler` over pinned source snapshots, including what it produced and the output hashes for reproducibility.

Example:

```yaml
id: compilation_run_wellness_2026_07_01
compiler_id: wellness_rule_compiler
compiler_version: "0.1.0"
source_snapshot_refs:
  - gwas_catalog_snapshot_2026_07
  - dbsnp_snapshot_2026_07
generated_record_refs:
  - rule_candidate_aldh2_rs671_alcohol_metabolism_001
  - evidence_candidate_aldh2_rs671_gwas_001
output_hashes:
  rule_set: "sha256:..."
  evidence_set: "sha256:..."
```

## 18e. ValidationReport

A `ValidationReport` records the results of strong validation over a compilation run's output. `status` is `pass` only when every validation dimension passes.

Example:

```yaml
id: validation_report_wellness_2026_07_01
compilation_run_ref: compilation_run_wellness_2026_07_01
results:
  schema_validation: pass
  evidence_validation: pass
  safety_validation: pass
  compatibility_validation: pass
  regression_validation: pass
status: pass
```

## 18f. DiffReport

A `DiffReport` compares a candidate release against the current active release and classifies each entry so review can focus on what changed.

Example:

```yaml
id: diff_report_wellness_v0_1_2026_07
compared_to_release: wellness_release_v0_1_2026_06
entries:
  - record_ref: aldh2_rs671_alcohol_metabolism
    classification: unchanged
  - record_ref: evidence_aldh2_rs671_001
    classification: expected_changed
  - record_ref: rule_candidate_new_trait_001
    classification: exception_changed
```

Classifications:

```text
unchanged          - identical to the active release
expected_changed   - changed in a way covered by validation, no per-item review needed
exception_changed  - triggers an ExceptionReview
```

## 18g. ExceptionReview

An `ExceptionReview` is the mandatory per-item human review, but only for exceptions and safety-sensitive diffs, not for every generated record. It reviews exceptions, not every rule.

Example:

```yaml
id: exception_review_wellness_2026_07_01
diff_report_ref: diff_report_wellness_v0_1_2026_07
reviewed_categories:
  - additions
  - deletions
  - conflicts
  - low_confidence
  - source_drift
  - validation_warning
  - safety_sensitive_change
items:
  - record_ref: rule_candidate_new_trait_001
    category: additions
    status: approved
    reviewed_by: curator_a
    reviewed_at: "2026-07-02"
status: approved
```

An `ExceptionReview` reviews only these categories:

```text
additions
deletions
conflicts
low_confidence
source_drift
validation_warning
safety_sensitive_change
```

A release cannot be approved while any `ExceptionReview` item is unresolved or rejected.

## 18h. ReleaseApproval

A `ReleaseApproval` is the mandatory human sign-off on the release bundle (the `RuntimeReleaseManifest`), not on each rule. It is the release-approval floor.

Example:

```yaml
id: release_approval_wellness_v0_1_2026_07
release_id: wellness_release_v0_1_2026_07
approved_by: governance_board
approved_at: "2026-07-07"
```

A `ReleaseApproval` approves the manifest as a whole; it presupposes passing validation reports, a diff report, and resolved exception reviews. Without a resolvable `ReleaseApproval`, the manifest cannot be active.

## 21. AlleleDefinitionRecord

Future PGx object.

## 27. Domain Model Validation Rules

Validation rules:

```text
1. Active RuleRecord must reference at least one active EvidenceRecord.
2. Active EvidenceRecord must reference at least one SourceRecord.
3. Candidate records must not be used by runtime Rule Engine.
4. RuleRecord interpretation_level must be educational in v0.1.
5. EvidenceRecord allowed_use must include educational_explanation.
6. EvidenceRecord disallowed_use must include blocked advice categories.
7. TraitInterpretationResult must include evidence_refs.
8. ExplanationRecord must include required safety message.
9. OutputSafetyValidationResult must pass before user-facing response.
10. RuleSetRelease cannot be active unless review_status is approved and test_status is passed.
```
