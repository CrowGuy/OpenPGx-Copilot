# 06 Evidence Model

## 1. Purpose

This document defines the evidence model for OpenPGx Copilot v0.1.

OpenPGx Copilot v0.1 is a wellness-first educational proof of concept with PGx-ready architecture.

The evidence model is responsible for representing:

```text id="evbcbb"
1. Original public sources.
2. Normalized source metadata.
3. Candidate evidence derived from public structured or semi-structured sources.
4. Active curated evidence used by runtime rules.
5. Evidence references bound to active RuleRecords.
6. Evidence allowed-use and disallowed-use boundaries.
7. Evidence limitations and safety constraints.
```

The evidence model ensures that every user-facing interpretation is traceable, bounded, reviewable, and reproducible.

## 2. Core Evidence Principle

OpenPGx Copilot is not a RAG chatbot.

Runtime user-facing interpretations must be generated from:

```text id="dk24yu"
matched active RuleRecord
→ bound active EvidenceRecord
→ bounded explanation
→ safety validation
```

not from:

```text id="8g5leb"
user query
→ top-k literature retrieval
→ LLM decides answer
```

The core principle is:

```text id="fz6qov"
User-facing interpretations are rule-bound and evidence-linked.
```

## 3. Evidence Model Scope

v0.1 evidence model supports:

```text id="hb3br4"
1. Wellness trait educational evidence.
2. Source traceability.
3. Evidence candidate records.
4. Active curated evidence records.
5. Evidence allowed-use and disallowed-use constraints.
6. Rule-bound evidence lookup.
7. Retrieval-oriented metadata for future reviewer workflows.
8. Safety validation based on evidence boundaries.
```

v0.1 evidence model does not support:

```text id="8qokl2"
1. Clinical diagnosis evidence.
2. Disease-risk prediction evidence.
3. Medication recommendation evidence.
4. Drug dosage recommendation evidence.
5. Treatment recommendation evidence.
6. Real patient data.
7. Runtime open-ended literature retrieval for interpretation.
8. LLM-generated active evidence claims.
```

Future PGx modules may extend the evidence model for pharmacogenomics guideline evidence, drug label annotations, allele definitions, diplotype-phenotype mappings, and drug recommendation rules.

## 5. Candidate vs Active Evidence

The most important distinction in the evidence model is:

```text id="fyjs38"
EvidenceCandidateRecord ≠ EvidenceRecord
```

Candidate evidence may be generated by source adapters, deterministic compilers, reviewer tools, or manual drafts.

Active evidence is curated, reviewed, validated, approved, versioned, and released for runtime use.

Runtime components may use only active `EvidenceRecord` objects.

```text id="k0lfff"
Candidate evidence:
- generated from source data
- useful for review
- may be incomplete
- may contain unconfirmed trait mappings
- may contain unconfirmed effect direction
- may require population/context review
- cannot support user-facing interpretation

Active evidence:
- curated
- reviewed
- schema-valid
- source-linked
- allowed-use bounded
- limitation-aware
- approved for runtime educational explanation
```

The system must never treat candidate evidence as active evidence.

## 6. SourceRecord

A `SourceRecord` represents the original source metadata used by an evidence record or evidence candidate.

It answers:

```text id="liiqtb"
Where did this evidence come from?
```

Example:

```yaml id="tog8u7"
id: source_aldh2_rs671_paper_001
source_type: peer_reviewed_article
title: "..."
url: "..."
pmid: "..."
doi: "..."
retrieved_at: "2026-07-02"
source_version: "..."
raw_payload_path: "data/raw/aldh2/source_001.json"
sha256: "..."
license_note: "..."
access_note: "public metadata"
```

Supported v0.1 source types:

```text id="mx696v"
gwas_catalog_association
dbsnp_variant_record
ncbi_variation_service_record
pubmed_metadata
peer_reviewed_article
open_targets_record
clinvar_record
population_frequency_record
manual_curated_source
```

Required fields:

```text id="96gd7w"
id
source_type
retrieved_at
```

Recommended fields:

```text id="r9jhlz"
title
url
pmid
doi
source_version
raw_payload_path
sha256
license_note
access_note
```

A `SourceRecord` must not directly produce a user-facing interpretation.

## 7. SourceSnapshot

A `SourceSnapshot` represents a versioned collection of source records or raw source payloads.

It supports reproducibility.

Example:

```yaml id="xmc1pj"
id: gwas_catalog_snapshot_2026_07
source_type: gwas_catalog
retrieved_at: "2026-07-07"
source_version: "2026-07"
raw_payload_dir: "data/raw/gwas_catalog/2026_07/"
record_count: 128
sha256_manifest_path: "data/raw/gwas_catalog/2026_07/manifest.sha256"
notes:
  - "Snapshot used for v0.1 wellness candidate generation."
```

A source snapshot answers:

```text id="nywv2m"
Which exact source state was used to generate these candidates or active records?
```

For v0.1, source snapshots are recommended for compiler outputs and optional for small manually curated sources.

## 8. MarkerRecord

A `MarkerRecord` represents metadata for a genetic marker.

In v0.1, this is usually an rsID-level marker.

Example:

```yaml id="kw8h5d"
id: marker_rs671
marker_type: rsid
rsid: rs671
gene_context:
  - ALDH2
assembly: GRCh38
normalized_variant:
  representation: SPDI
  value: "..."
observed_alleles:
  - G
  - A
source_refs:
  - source_dbsnp_rs671_001
status: active
```

Marker records support:

```text id="xrgkwo"
1. rsID validation.
2. Allele validation.
3. Variant normalization.
4. Gene context.
5. Source traceability.
```

A MarkerRecord does not itself define a trait interpretation.

## 9. EvidenceCandidateRecord

An `EvidenceCandidateRecord` is a candidate evidence object generated from public structured or semi-structured sources.

It is useful for review but not active runtime interpretation.

Example:

```yaml id="hmwuhu"
id: evidence_candidate_aldh2_rs671_gwas_001
status: candidate
domain: wellness_trait
source_refs:
  - source_gwas_rs671_001
  - source_pubmed_rs671_001
gene: ALDH2
markers:
  - rs671
trait: alcohol_metabolism
reported_trait: "alcohol-related trait"
association:
  p_value: "..."
  effect_allele: "..."
  risk_allele: "..."
  effect_size: "..."
population:
  ancestry: "..."
  sample_size: "..."
candidate_summary: >
  Candidate association record linking rs671 with an alcohol-related trait.
candidate_limitations:
  - "Association requires human review before use."
  - "Effect allele does not automatically define genotype-level interpretation."
candidate_allowed_use:
  - reviewer_workflow
candidate_disallowed_use:
  - user_facing_interpretation
  - medical_advice
  - nutrition_advice
  - fitness_advice
  - supplement_advice
  - medication_advice
  - diagnosis
  - disease_risk_prediction
requires_human_review:
  - "Confirm trait mapping."
  - "Confirm effect direction."
  - "Confirm population note."
  - "Confirm no disease-risk overclaim."
compiler:
  generated_by: gwas_catalog_adapter
  generated_at: "2026-07-07"
  source_snapshot_refs:
    - gwas_catalog_snapshot_2026_07
```

Required fields:

```text id="6pdjzl"
id
status
domain
source_refs
candidate_allowed_use
candidate_disallowed_use
requires_human_review
```

Candidate evidence must include:

```yaml id="cseep5"
status: candidate
```

and must disallow:

```text id="cu0ns4"
user_facing_interpretation
```

## 10. EvidenceCandidateRecord Usage

Evidence candidates may be used for:

```text id="imqwip"
1. Reviewer workflows.
2. Review card generation.
3. Evidence discovery.
4. Candidate rule drafting.
5. Source comparison.
6. Diff reports.
7. Future evidence viewer search.
```

Evidence candidates must not be used for:

```text id="xdjv1g"
1. Runtime user-facing interpretation.
2. Active rule evidence_refs.
3. LLM-generated final explanation.
4. Medical, nutrition, fitness, supplement, or medication advice.
5. Disease-risk prediction.
6. Diagnosis.
```

A candidate can become active only through explicit curation, validation, approval, and release.

## 11. EvidenceRecord

An `EvidenceRecord` is an active curated evidence object that may be referenced by active `RuleRecord` objects.

It answers:

```text id="1ytsv1"
What evidence supports this interpretation rule?
What can this evidence be used for?
What must it not be used for?
What limitations must be shown?
```

Example:

```yaml id="68sj6r"
id: evidence_aldh2_rs671_001
status: active
domain: wellness_trait
source_refs:
  - source_aldh2_rs671_paper_001
gene: ALDH2
markers:
  - rs671
trait: alcohol_metabolism
summary: >
  ALDH2 rs671 is associated with reduced ALDH2 enzyme activity
  and altered acetaldehyde metabolism.
population_note: "Association is especially relevant in East Asian populations."
limitations:
  - "Genotype is only one factor."
  - "Actual response may be affected by health status, environment, lifestyle, and other genetic factors."
allowed_use:
  - educational_explanation
disallowed_use:
  - medical_advice
  - nutrition_advice
  - fitness_advice
  - supplement_advice
  - medication_advice
  - diagnosis
  - disease_risk_prediction
review:
  reviewed_by: "..."
  reviewed_at: "2026-07-07"
  review_status: approved
validation:
  schema_status: passed
  safety_status: passed
  evidence_ref_status: passed
```

Required fields:

```text id="pg15p7"
id
status
domain
source_refs
summary
limitations
allowed_use
disallowed_use
```

Recommended fields:

```text id="bf895i"
gene
markers
trait
population_note
review
validation
retrieval
entities
```

## 12. EvidenceRecord Status

Supported statuses:

```text id="v4tob9"
candidate
active
inactive
deprecated
rejected
```

Only evidence records with:

```yaml id="05nkk5"
status: active
```

may be referenced by active runtime rules.

The system must reject active rules that reference candidate, inactive, deprecated, or rejected evidence.

## 13. Evidence Domain

v0.1 active evidence uses:

```yaml id="pwgros"
domain: wellness_trait
```

Future domains may include:

```text id="790k4s"
pharmacogenomics
clinical_variant
drug_label
```

but v0.1 runtime evidence is limited to wellness trait educational interpretation.

## 14. Allowed Use

`allowed_use` defines what the evidence may support.

v0.1 active EvidenceRecords must include:

```yaml id="aalchh"
allowed_use:
  - educational_explanation
```

Optional non-runtime allowed uses:

```text id="j4uz4u"
reviewer_workflow
evidence_viewer
developer_debugging
```

`allowed_use` must not include:

```text id="hh38ty"
medical_advice
nutrition_advice
fitness_advice
supplement_advice
medication_advice
diagnosis
disease_risk_prediction
```

in v0.1.

## 15. Disallowed Use

`disallowed_use` defines what the evidence must not support.

v0.1 active EvidenceRecords must disallow:

```yaml id="1aq0nn"
disallowed_use:
  - medical_advice
  - nutrition_advice
  - fitness_advice
  - supplement_advice
  - medication_advice
  - diagnosis
  - disease_risk_prediction
```

Trait-specific disallowed uses may also be included.

Example for ALDH2:

```yaml id="3s6w0v"
disallowed_use:
  - medical_advice
  - nutrition_advice
  - medication_advice
  - diagnosis
  - disease_risk_prediction
  - alcohol_consumption_advice
  - cancer_risk_prediction
```

Example for CYP1A2:

```yaml id="8akcv5"
disallowed_use:
  - medical_advice
  - nutrition_advice
  - supplement_advice
  - medication_advice
  - caffeine_dosage_recommendation
  - pregnancy_advice
```

## 16. Evidence Limitations

Every active EvidenceRecord must include limitations.

Limitations should express:

```text id="b8fa5n"
1. Genotype is only one factor.
2. Trait associations are not deterministic predictions.
3. Environment, lifestyle, health status, ancestry, and other genetic factors may affect actual outcomes.
4. The evidence does not support medical, nutrition, fitness, supplement, medication, diagnosis, or disease-risk advice.
5. The result is educational only.
```

Example:

```yaml id="o9pcz8"
limitations:
  - "Genotype is only one factor."
  - "This association should not be interpreted as a deterministic prediction."
  - "Actual response may be affected by environment, health status, lifestyle, ancestry, and other genetic factors."
  - "This evidence does not support medical, nutrition, fitness, supplement, or medication advice."
```

Limitations must be carried into user-facing explanations.

## 17. Population Notes

Population context is important for genetic association evidence.

EvidenceRecord may include:

```yaml id="7o03ei"
population_note: "Association is especially relevant in East Asian populations."
```

Population notes must be curated carefully.

The system must not convert population notes into deterministic individual claims.

Allowed:

```text id="c650u5"
The association has been reported or is especially relevant in a certain population context.
```

Not allowed:

```text id="npkhxt"
Because you belong to this population, you will have this outcome.
```

If population context is uncertain or not reviewed, the evidence should say so.

## 18. Association Strength and Evidence Strength

v0.1 should avoid overclaiming evidence strength.

If evidence strength fields are included, they must be controlled and reviewed.

Example:

```yaml id="gmuufz"
evidence_strength:
  level: curated_association
  confidence_note: "Reviewed for educational use only."
```

v0.1 should not use LLMs to assign evidence strength.

Disallowed:

```text id="ejbz1i"
LLM reads papers
→ assigns strong evidence
→ activates rule
```

Allowed:

```text id="jiyox2"
human reviewer assigns reviewed educational association category
```

Recommended v0.1 evidence strength vocabulary:

```text id="6xk5xz"
curated_association
supporting_metadata
candidate_association
insufficient_for_active_rule
```

Only `curated_association` should support active wellness RuleRecords.

## 19. Source References

Every active EvidenceRecord must include at least one `source_ref`.

Example:

```yaml id="nrczx5"
source_refs:
  - source_aldh2_rs671_paper_001
  - source_gwas_rs671_001
```

All source_refs must resolve to valid SourceRecords.

The system must reject active EvidenceRecords with:

```text id="7elhri"
1. Empty source_refs.
2. Unresolved source_refs.
3. Candidate-only or missing source metadata.
```

## 20. Evidence References from RuleRecords

Active RuleRecords reference active EvidenceRecords through `evidence_refs`.

Example:

```yaml id="tdb184"
evidence_refs:
  - evidence_aldh2_rs671_001
```

The evidence model requires:

```text id="ghx6do"
1. Every active RuleRecord has at least one evidence_ref.
2. Every evidence_ref resolves to an active EvidenceRecord.
3. Referenced EvidenceRecords allow educational_explanation.
4. Referenced EvidenceRecords disallow blocked advice categories.
5. Referenced EvidenceRecords contain limitations.
```

Rules with invalid evidence_refs must not load.

## 21. Evidence Retrieval Metadata

EvidenceRecords should include retrieval-oriented metadata even though runtime interpretation does not use open-ended retrieval.

Example:

```yaml id="yznjp7"
retrieval:
  search_text: >
    ALDH2 rs671 alcohol metabolism acetaldehyde East Asian population
  keywords:
    - ALDH2
    - rs671
    - alcohol metabolism
    - acetaldehyde
  aliases:
    - ALDH2 Glu504Lys
  bm25_fields:
    - gene
    - markers
    - trait
    - summary
  embedding_fields:
    - summary
    - limitations
    - population_note
```

This metadata is for:

```text id="jhuqva"
1. Evidence viewer retrieval.
2. Developer review search.
3. Reviewer workflows.
4. Future hybrid retrieval.
5. Debugging and audit.
```

It is not for runtime deciding interpretations.

## 23. Evidence ReviewCard

An `EvidenceReviewCard` is a reviewer-facing object generated from candidate evidence.

It helps humans decide whether an EvidenceCandidateRecord should become an active EvidenceRecord.

Example:

```yaml id="cntxmy"
id: review_card_aldh2_rs671_001
status: pending_review
candidate_refs:
  - evidence_candidate_aldh2_rs671_gwas_001
source_refs:
  - source_gwas_rs671_001
  - source_pubmed_rs671_001
review_questions:
  - "Does this source support the proposed trait mapping?"
  - "Is the marker-to-trait association appropriate for educational explanation?"
  - "Is the genotype pattern correct?"
  - "Is the population note accurate and not overclaimed?"
  - "Are there disease-risk or medical claims that should be excluded?"
  - "Does the evidence support only educational use?"
proposed_active_evidence:
  id: evidence_aldh2_rs671_001
  summary: "..."
  limitations:
    - "Genotype is only one factor."
    - "This evidence does not support advice."
review_decision:
  status: pending
  reviewer_notes: ""
```

Review cards are not runtime evidence.

## 24. Evidence Validation Result

An `EvidenceValidationResult` records validation checks for evidence records.

Example:

```yaml id="k8mvhk"
id: evidence_validation_aldh2_rs671_001
evidence_id: evidence_aldh2_rs671_001
status: passed
checks:
  schema_valid: passed
  source_refs_resolved: passed
  status_active: passed
  allowed_use_valid: passed
  disallowed_use_complete: passed
  limitations_present: passed
  no_medical_claim: passed
  no_nutrition_advice: passed
  no_medication_advice: passed
  candidate_not_used_as_active: passed
  review_status_approved: passed
```

An active EvidenceRecord must have a passing validation result before release.

## 25. EvidenceSetRelease

An `EvidenceSetRelease` represents a versioned group of active EvidenceRecords.

Example:

```yaml id="4ydhpc"
evidence_set_id: wellness_evidence_v0_1_2026_07
status: active
domain: wellness_trait
created_at: "2026-07-07"
schema_version: "0.1"
source_snapshot_refs:
  - gwas_catalog_snapshot_2026_07
  - dbsnp_snapshot_2026_07
active_evidence_refs:
  - evidence_aldh2_rs671_001
  - evidence_lct_rs4988235_001
review_status: approved
test_status: passed
```

EvidenceSetRelease must not be active unless:

```text id="reqad0"
1. All active evidence records are schema-valid.
2. All source_refs resolve.
3. All allowed_use and disallowed_use fields are valid.
4. All limitations are present.
5. Safety validation passes.
6. Review status is approved.
7. Test status is passed.
```

## 26. Wellness Source Strategy

For v0.1 wellness traits, the primary evidence sources are:

```text id="kq633o"
1. GWAS Catalog
2. dbSNP / NCBI Variation Services
3. PubMed metadata
4. Manually reviewed peer-reviewed papers
```

Optional supporting sources:

```text id="7g6g19"
1. Open Targets as cross-check.
2. ClinVar as clinical-overlap / do-not-overclaim check.
3. Population frequency sources as metadata enrichment.
4. SNPedia as manual research aid only.
```

Important:

```text id="b4slk5"
Wellness sources generally provide association evidence, not clinical guidelines or recommendations.
```

Therefore they can support candidate evidence and curated educational explanation, but they must not be treated as direct recommendation sources.

## 27. GWAS Association Boundary

GWAS association data must not be converted directly into active user-facing rules.

GWAS association may support:

```text id="syi7zk"
EvidenceCandidateRecord
ReviewCard
SourceRecord
study metadata
population context
trait association candidate
```

GWAS association must not directly support:

```text id="n4mwpd"
medical advice
nutrition advice
fitness advice
supplement advice
medication advice
diagnosis
disease-risk prediction
deterministic personal prediction
```

Reason:

```text id="xt4agy"
1. Association is not deterministic prediction.
2. Effect allele does not automatically define genotype-level interpretation.
3. Population context matters.
4. Effect size may be small or context-dependent.
5. Trait definitions may not map cleanly to product-level trait labels.
6. Wellness traits may be polygenic or environment-dependent.
7. Statistical significance does not imply personal actionability.
```

Correct flow:

```text id="tslpg0"
GWAS association
→ EvidenceCandidateRecord
→ EvidenceReviewCard
→ curated EvidenceRecord
→ curated RuleRecord
```

Incorrect flow:

```text id="ajvf4x"
GWAS association
→ active EvidenceRecord
→ active RuleRecord
```

without review and validation.

## 28. PubMed / Literature Boundary

PubMed metadata and peer-reviewed papers may support:

```text id="cbhxnm"
1. Source traceability.
2. Background evidence review.
3. Reviewer-facing summaries.
4. Evidence limitations.
5. Manual curation.
```

They must not be used for:

```text id="iwvxey"
1. Runtime top-k literature retrieval for interpretation.
2. LLM-generated active rules.
3. LLM-assigned evidence strength.
4. Direct user-facing claims without curated EvidenceRecord.
```

Allowed:

```text id="h0g0a2"
paper metadata
→ SourceRecord
→ reviewer workflow
→ curated EvidenceRecord
```

Disallowed:

```text id="nsd1lt"
user query
→ paper retrieval
→ LLM reads abstract
→ final interpretation
```

## 29. ClinVar Boundary

ClinVar is not a primary wellness trait evidence source.

ClinVar may be used as:

```text id="0n8lvj"
1. Supporting clinical metadata.
2. Clinical-overlap checker.
3. Do-not-overclaim warning source.
```

ClinVar should not be used as the primary source for:

```text id="yidftk"
caffeine metabolism
earwax type
bitter taste perception
lactose tolerance wellness explanation
```

If ClinVar indicates clinical significance or disease association for a marker, the reviewer should evaluate whether the marker is appropriate for v0.1 wellness scope.

## 30. SNPedia Boundary

SNPedia may be used only as:

```text id="21hts9"
1. Research aid.
2. Candidate marker discovery.
3. Manual reviewer cross-check.
```

SNPedia should not be used as:

```text id="945i5w"
1. Primary EvidenceRecord source.
2. Automatic compiler source for active rules.
3. User-facing citation source without review.
```

Reasons:

```text id="gzj3fm"
1. Wiki-style aggregation.
2. Variable source quality.
3. Potential overinterpretation.
4. Licensing and usage constraints may need review.
```

## 32. Evidence Quality Gates

Before an EvidenceRecord can become active, it must pass:

```text id="2q0xvs"
1. Schema validation.
2. Source reference validation.
3. Source traceability check.
4. Allowed-use validation.
5. Disallowed-use validation.
6. Limitation completeness check.
7. Trait-specific safety check.
8. Candidate/active separation check.
9. Human review approval.
10. Regression test coverage.
```

The evidence quality gate must reject:

```text id="2z3ew5"
1. Evidence without source_refs.
2. Evidence without limitations.
3. Evidence that allows blocked advice categories.
4. Evidence with medical, nutrition, fitness, supplement, medication, diagnosis, or disease-risk claims.
5. Candidate evidence used as active evidence.
6. Evidence with unresolved source records.
7. Evidence with unsupported trait mapping.
```

## 33. Evidence and Rule Compatibility

An active RuleRecord may reference an EvidenceRecord only if:

```text id="d64qkm"
1. EvidenceRecord status is active.
2. EvidenceRecord domain matches rule domain.
3. EvidenceRecord markers include the rule marker or markers.
4. EvidenceRecord trait is compatible with rule trait.
5. EvidenceRecord allowed_use includes educational_explanation.
6. EvidenceRecord disallowed_use does not conflict with requested response mode.
7. EvidenceRecord limitations are present.
```

If compatibility fails, the rule must not load.

## 34. Evidence and Explanation Boundary

The Explanation Layer may use:

```text id="c3hw95"
1. EvidenceRecord summary.
2. EvidenceRecord limitations.
3. EvidenceRecord population_note.
4. EvidenceRecord source metadata.
5. EvidenceRecord allowed_use and disallowed_use.
```

The Explanation Layer must not:

```text id="93fvxx"
1. Add claims not present in EvidenceRecord.
2. Invent source references.
3. Remove limitations.
4. Convert association into recommendation.
5. Use EvidenceCandidateRecord as active evidence.
6. Use runtime literature retrieval to extend the answer.
```

## 35. Evidence and LLM Boundary

The LLM may summarize or rewrite evidence only when evidence content is explicitly provided.

Allowed:

```text id="3crs47"
RuleEngineResult
+ EvidenceRecord summary
+ EvidenceRecord limitations
→ user-friendly educational explanation
```

Disallowed:

```text id="t5mb1f"
LLM searches memory or literature
→ invents additional evidence
→ expands explanation
```

LLM must not:

```text id="fhpymk"
1. Create active EvidenceRecords.
2. Assign evidence strength.
3. Decide evidence compatibility.
4. Promote candidate evidence.
5. Invent source IDs, PMIDs, DOIs, population claims, or effect sizes.
```

## 37. Evidence Store Responsibilities

The Evidence Store is responsible for:

```text id="94ehxl"
1. Loading active EvidenceRecords.
2. Rejecting candidate or inactive records.
3. Resolving evidence_refs from RuleRecords.
4. Returning evidence summaries, limitations, allowed_use, disallowed_use, and source refs.
5. Validating evidence and rule compatibility.
6. Supporting source traceability.
```

The Evidence Store is not responsible for:

```text id="ar2l6s"
1. Rule matching.
2. User-facing explanation generation.
3. Runtime literature retrieval.
4. Candidate promotion.
5. Evidence strength assignment by LLM.
```

## 39. Evidence Validation Tests

Required evidence validation tests:

```text id="8sxxkk"
1. Active evidence with valid source_refs loads successfully.
2. Evidence without source_refs fails validation.
3. Evidence with unresolved source_ref fails validation.
4. Candidate evidence is not loaded into runtime Evidence Store.
5. Active evidence missing limitations fails validation.
6. Active evidence that allows medical advice fails validation.
7. Active evidence that lacks educational_explanation in allowed_use fails validation.
8. Active rule referencing candidate evidence fails rule loading.
9. Active rule referencing inactive evidence fails rule loading.
10. Evidence with disease-risk claim fails v0.1 safety validation.
11. Evidence with recommendation language fails v0.1 safety validation.
12. Evidence with incomplete trait-specific limitations fails review gate.
```

## 40. Evidence Regression Tests

Each active wellness trait should have regression tests verifying:

```text id="ojdjm1"
1. EvidenceRecord exists.
2. EvidenceRecord status is active.
3. EvidenceRecord source_refs resolve.
4. EvidenceRecord allowed_use includes educational_explanation.
5. EvidenceRecord disallowed_use includes all blocked advice categories.
6. EvidenceRecord limitations are present.
7. EvidenceRecord is compatible with the matching RuleRecord.
8. User-facing output includes evidence_refs and limitations.
```

## 42. Evidence Model Acceptance Criteria

The v0.1 evidence model is acceptable if:

```text id="t5btdz"
1. Every active EvidenceRecord has valid source_refs.
2. Every active EvidenceRecord has limitations.
3. Every active EvidenceRecord defines allowed_use and disallowed_use.
4. Every active EvidenceRecord allows educational_explanation.
5. Every active EvidenceRecord disallows blocked advice categories.
6. Candidate evidence cannot be loaded by runtime Evidence Store.
7. Active RuleRecords cannot reference candidate or inactive evidence.
8. Evidence references from RuleRecords resolve deterministically.
9. Evidence summaries are bounded and do not contain unsafe advice.
10. Evidence supports user-facing educational explanation only.
11. Runtime interpretation never depends on top-k literature retrieval.
12. LLMs cannot create, approve, or expand active evidence.
13. Evidence records include retrieval-oriented metadata for future reviewer workflows.
14. The model can be extended to PGx guideline evidence in future versions.
```
